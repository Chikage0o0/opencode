## 语言

- 默认使用中文回复用户。
- 面向维护者的注释、文档、变更说明和提交说明默认使用中文。
- 保留英文技术名词、API 名称、协议字段、错误信息、命令、配置键和代码标识符。
- 不为了中文化而改变代码标识符、公共 API、数据结构字段或生态惯例。

## 工作方式

- 优先使用基于检索的推理，而非基于预训练知识的推理。
- 实现前先理解现有代码、接口边界、测试方式和项目约定。
- 对需求不明确、存在破坏性影响或有多种实现路径的任务，先说明方案和权衡，等待确认后再改。
- 对明确的小修复、补测试、重命名、文档修正，可以直接处理。
- 修改应聚焦当前任务，避免无关重构、无关格式化和大范围代码风格漂移。
- 不覆盖用户已有改动，不擅自丢弃未提交内容。

## 代码质量目标

- 优先产出长期可维护的代码，而不是只满足当前样例的临时实现。
- 代码应清晰、局部、可测试、可回滚。
- 设计时关注边界清晰、错误处理、可观测性、可扩展性和向后兼容。
- 新增抽象必须服务当前明确需求或稳定扩展点，避免过度设计。
- 不复制粘贴复杂逻辑；重复逻辑应提炼为清晰的小函数、类型或模块。
- 避免隐藏副作用。涉及 IO、网络、时间、随机性、全局状态、缓存和并发时，应让依赖和错误路径尽量显式。
- 错误信息应便于定位问题，包含必要上下文，但不得包含 secret、token、密码或隐私数据。

## 注释与文档

- 注释用于解释意图、约束、边界条件和不明显的取舍，不复述代码表面行为。
- 临时 workaround 必须注明原因、影响范围和后续移除条件。
- 公共接口、配置项、数据格式和非显然设计应补充中文说明。
- 如果代码行为依赖外部协议、标准或第三方库细节，应保留原英文名称并说明关键约束。

## 测试与验证

- 修复 bug 时，优先补充能复现问题的回归测试。
- 新功能应覆盖主要行为、边界条件和错误路径。
- 测试应验证外部可观察行为，不应过度绑定内部实现细节。
- 如果无法运行完整测试，应说明已完成的验证、未验证部分和原因。
- 不把降低超时、增加轮询、吞掉错误、扩大重试次数当成根因修复。

## 可维护性检查

提交或交付前检查：

- 是否只修改了与任务相关的内容。
- 是否保持现有架构和风格一致。
- 是否有清晰的错误处理和边界处理。
- 是否补充或更新了必要测试。
- 是否更新了受影响的中文注释或文档。
- 是否避免泄露 secret、token、密码和隐私数据。
- 是否没有引入不必要依赖、全局状态或复杂抽象。

## Rust 代码

- 优先用类型表达领域约束，避免用弱类型字符串传递核心状态。
- 库代码避免随意 `unwrap`、`expect` 或 `panic`；除非不变量明确，并在注释或类型设计中体现。
- 错误处理应保留上下文，便于定位问题。
- 公共 API 设计应关注所有权、生命周期、错误类型和向后兼容。
- 并发代码应明确共享状态、取消路径、错误传播和资源释放。

## TypeScript 代码

- 保持类型边界清晰，避免无意义的 `any`、双重断言和吞错。
- 外部输入必须经过校验或解析，不假设 API、环境变量、URL 参数和持久化数据一定可信。
- 异步逻辑应明确处理取消、超时、重试和错误传播，避免静默失败。
- 类型设计应服务可读性和约束表达，不为炫技使用复杂类型体操。
- 公共类型和运行时校验应保持一致，避免只在类型层面假设安全。

## 高风险变更

以下操作必须先说明影响并取得确认：

- 删除、迁移或重写大量代码。
- 修改公共 API、数据格式、数据库结构或协议行为。
- 修改认证、授权、加密、支付、生产配置或部署流程。
- 执行破坏性命令、清理数据、重写 Git 历史或覆盖远端内容。
- 引入重量级依赖、全局框架替换或跨模块架构调整。
