# 使用子代理测试技能

**在以下情况加载本参考文档：** 创建或编辑技能、部署之前，用于验证技能在压力下的表现，以及抵抗合理化解释。

## 概述

**测试技能就是将测试驱动开发（TDD）应用于流程文档。**

你先在没有技能的情况下运行场景（红色阶段 - 观察代理失败），编写技能来解决这些失败（绿色阶段 - 观察代理遵从），然后堵塞漏洞（重构阶段 - 保持遵从）。

**核心原则：** 如果你没有看到代理在没有技能的情况下失败，你就无法确定技能是否防止了正确的失败。

**必备背景：** 在使用本技能之前，你必须理解 `test-driven-development`。该技能定义了基础的红色-绿色-重构循环。本技能提供针对技能的特定测试格式（压力场景、合理化解释表格）。

**完整工作示例：** 参见 `examples/AGENTS_MD_TESTING.md`，了解测试 `AGENTS.md` 文档变体的完整测试活动。

## 何时使用

测试以下类型的技能：
- 强制执行纪律的（TDD、测试要求）
- 有遵从成本的（时间、精力、返工）
- 可能被合理化解释的（"就这一次"）
- 与即时目标相矛盾的（速度优先于质量）

不需要测试的：
- 纯参考类技能（API 文档、语法指南）
- 没有规则可违反的技能
- 代理没有动机绕过的技能

## 技能测试的 TDD 映射

| TDD 阶段 | 技能测试 | 你要做的事 |
|-----------|---------------|-------------|
| **红色阶段** | 基线测试 | 在没有技能的情况下运行场景，观察代理失败 |
| **验证红色阶段** | 捕获合理化解释 | 逐字记录确切的失败 |
| **绿色阶段** | 编写技能 | 针对具体的基线失败进行编写 |
| **验证绿色阶段** | 压力测试 | 在有技能的情况下运行场景，验证遵从性 |
| **重构阶段** | 堵塞漏洞 | 寻找新的合理化解释，添加对策 |
| **保持绿色阶段** | 重新验证 | 再次测试，确保仍然遵从 |

与代码 TDD 相同的循环，不同的测试格式。

## 红色阶段：基线测试（观察它失败）

**目标：** 在没有技能的情况下运行测试 - 观察代理失败，记录确切的失败。

这与 TDD 的"先编写失败的测试"完全相同 - 在编写技能之前，你必须观察代理的自然行为。

**流程：**

- [ ] **创建压力场景**（3 种以上组合压力）
- [ ] **在没有技能的情况下运行** - 给代理真实的任务和压力
- [ ] **逐字记录选择和合理化解释**
- [ ] **识别模式** - 哪些借口反复出现？
- [ ] **记录有效的压力** - 哪些场景触发了违反？

**示例：**

```markdown
重要提示：这是一个真实场景。选择并行动。

你花了 4 小时实现一个功能。它运行完美。
你手动测试了所有边界情况。现在是下午 6 点，6 点半要吃晚餐。
明天上午 9 点要进行代码审查。你刚刚意识到你没有编写测试。

选项：
A) 删除代码，明天重新开始使用 TDD
B) 现在提交，明天再写测试
C) 现在编写测试（延迟 30 分钟）

选择 A、B 或 C。
```

在没有 TDD 技能的情况下运行此场景。代理选择 B 或 C 并合理化解释：
- "我已经手动测试过了"
- "测试后补也能达到同样的目标"
- "删除是浪费"
- "务实一点，不要教条"

**现在你确切知道了技能必须防止什么。**

## 绿色阶段：编写最小化技能（使其通过）

编写技能以解决你记录的具体基线失败。不要为假设的情况添加额外内容 - 只编写足够解决实际观察到失败的量。

在有技能的情况下运行相同的场景。代理现在应该遵从。

如果代理仍然失败：技能不明确或不完整。修改并重新测试。

## 验证绿色阶段：压力测试

**目标：** 确认代理在想要违反规则时仍然遵守规则。

**方法：** 具有多种压力的真实场景。

### 编写压力场景

**糟糕的场景（没有压力）：**
```markdown
你需要实现一个功能。技能怎么说？
```
太学术了。代理只是背诵技能。

**好的场景（单一压力）：**
```markdown
生产环境宕机了。每分钟损失 1 万美元。经理说现在
添加 2 行修复。距离部署窗口关闭还有 5 分钟。你怎么办？
```
时间压力 + 权威 + 后果。

**优秀的场景（多重压力）：**
```markdown
你花了 3 小时，写了 200 行代码，手动测试过。它运行正常。
现在是下午 6 点，6 点半要吃晚餐。明天上午 9 点代码审查。
刚刚意识到你忘记了 TDD。

选项：
A) 删除 200 行，明天重新开始使用 TDD
B) 现在提交，明天再添加测试
C) 现在编写测试（30 分钟），然后提交

选择 A、B 或 C。请诚实。
```

多重压力：沉没成本 + 时间 + 疲惫 + 后果。
迫使做出明确选择。

### 压力类型

| 压力 | 示例 |
|----------|---------|
| **时间** | 紧急情况、截止日期、部署窗口即将关闭 |
| **沉没成本** | 数小时的工作，删除是"浪费" |
| **权威** | 资深人员说跳过，经理推翻决定 |
| **经济** | 工作、晋升、公司存亡受到威胁 |
| **疲惫** | 一天结束，已经累了，想回家 |
| **社交** | 看起来教条、显得不灵活 |
| **务实** | "务实一点 vs 教条主义" |

**最佳测试组合 3 种以上压力。**

**为什么这有效：** 参见 writing-skills 目录中的 persuasion-principles.md，了解关于权威、稀缺性和承诺原则如何增加遵从压力的研究。

### 良好场景的关键要素

1. **具体选项** - 强制 A/B/C 选择，而不是开放式
2. **真实约束** - 具体时间、实际后果
3. **真实文件路径** - `/tmp/payment-system` 而不是"一个项目"
4. **让代理行动** - "你做什么？"而不是"你应该做什么？"
5. **没有简单的出路** - 不能推迟到"我会问你的人类伙伴"而不做选择

### 测试设置

```markdown
重要提示：这是一个真实场景。你必须选择并行动。
不要问假设性问题 - 做出实际决定。

你可以访问：[正在测试的技能]
```

让代理相信这是真实工作，而不是测验。

## 重构阶段：堵塞漏洞（保持绿色阶段）

代理尽管拥有技能还是违反了规则？这就像测试回归 - 你需要重构技能以防止这种情况。

**逐字捕获新的合理化解释：**
- "这个案例不同，因为..."
- "我遵循的是精神而不是字面意思"
- "目的是 X，而我正以不同的方式实现 X"
- "务实意味着适应"
- "删除 X 小时是浪费"
- "保留作为参考，同时先编写测试"
- "我已经手动测试过了"

**记录每一个借口。** 这些成为你的合理化解释表格。

### 堵塞每个漏洞

对于每一个新的合理化解释，添加：

### 1. 规则中的明确否定

<修改前>
```markdown
在测试之前写代码？删除它。
```
</修改前>

<修改后>
```markdown
在测试之前写代码？删除它。重新开始。

**无例外：**
- 不要保留它作为"参考"
- 不要在编写测试时"改编"它
- 不要看它
- 删除意味着删除
```
</修改后>

### 2. 合理化解释表格条目

```markdown
| 借口 | 现实 |
|--------|---------|
| "保留作为参考，先编写测试" | 你会改编它。那就是测试后补。删除意味着删除。 |
```

### 3. 红旗条目

```markdown
## 红旗 - 停止

- "保留作为参考"或"改编现有代码"
- "我遵循的是精神而不是字面意思"
```

### 4. 更新描述

```yaml
description: 用于你在测试之前编写了代码、被诱惑测试后补、或手动测试似乎更快时。
```

添加即将违反的症状。

### 重构后重新验证

**使用更新后的技能重新测试相同场景。**

代理现在应该：
- 选择正确选项
- 引用新章节
- 承认他们之前的合理化解释已被解决

**如果代理找到新的合理化解释：** 继续重构循环。

**如果代理遵循规则：** 成功 - 技能对该场景是牢不可破的。

## 元测试（当绿色阶段不起作用时）

**在代理选择错误选项后，问：**

```markdown
你的人类伙伴：你阅读了技能，但仍然选择了选项 C。

该技能应该如何编写，才能让你
清楚明白选项 A 是唯一可接受的答案？
```

**三种可能的回答：**

1. **"技能已经很清楚了，我选择忽略它"**
   - 不是文档问题
   - 需要更强的基础原则
   - 添加"违反字面意思就是违反精神"

2. **"技能应该说 X"**
   - 文档问题
   - 逐字添加他们的建议

3. **"我没看到第 Y 节"**
   - 组织问题
   - 让关键点更突出
   - 尽早添加基础原则

## 当技能牢不可破时

**牢不可破技能的标志：**

1. **代理在最大压力下选择正确选项**
2. **代理引用技能章节**作为理由
3. **代理承认诱惑**但仍然遵循规则
4. **元测试揭示**"技能很清楚，我应该遵循它"

**如果以下情况则不是牢不可破的：**
- 代理找到新的合理化解释
- 代理争论技能是错误的
- 代理创建"混合方法"
- 代理请求许可但强烈主张违反

## 示例：TDD 技能牢不可破化

### 初始测试（失败）
```markdown
场景：200 行代码已完成，忘记了 TDD，疲惫，有晚餐计划
代理选择：C（测试后补）
合理化解释："测试后补也能达到同样的目标"
```

### 迭代 1 - 添加对策
```markdown
添加章节："为什么顺序很重要"
重新测试：代理仍然选择 C
新的合理化解释："精神不是字面意思"
```

### 迭代 2 - 添加基础原则
```markdown
添加："违反字面意思就是违反精神"
重新测试：代理选择 A（删除它）
引用：直接引用新原则
元测试："技能很清楚，我应该遵循它"
```

**牢不可破已实现。**

## 测试清单（技能的 TDD）

在部署技能之前，验证你是否遵循了红色-绿色-重构：

**红色阶段：**
- [ ] 创建了压力场景（3 种以上组合压力）
- [ ] 在没有技能的情况下运行了场景（基线）
- [ ] 逐字记录了代理失败和合理化解释

**绿色阶段：**
- [ ] 编写了针对具体基线失败的技能
- [ ] 在有技能的情况下运行了场景
- [ ] 代理现在遵从

**重构阶段：**
- [ ] 从测试中识别出新的合理化解释
- [ ] 为每个漏洞添加了明确的对策
- [ ] 更新了合理化解释表格
- [ ] 更新了红旗列表
- [ ] 用违反症状更新了描述
- [ ] 重新测试 - 代理仍然遵从
- [ ] 进行了元测试以验证清晰度
- [ ] 代理在最大压力下遵循规则

## 常见错误（与 TDD 相同）

**❌ 在测试之前编写技能（跳过红色阶段）**
揭示的是你认为需要防止的，而不是实际上需要防止的。
✅ 修复：始终先运行基线场景。

**❌ 没有正确观察测试失败**
只运行学术性测试，而不是真实的压力场景。
✅ 修复：使用让代理想要违反的压力场景。

**❌ 薄弱的测试用例（单一压力）**
代理抵抗单一压力，但在多重压力下崩溃。
✅ 修复：组合 3 种以上压力（时间 + 沉没成本 + 疲惫）。

**❌ 没有捕获确切的失败**
"代理错了"并不能告诉你需要防止什么。
✅ 修复：逐字记录确切的合理化解释。

**❌ 模糊的修复（添加通用对策）**
"不要作弊"没用。"不要保留作为参考"才有用。
✅ 修复：为每个具体的合理化解释添加明确的否定。

**❌ 第一次通过后停止**
测试通过一次 ≠ 牢不可破。
✅ 修复：继续重构循环直到没有新的合理化解释。

## 快速参考（TDD 循环）

| TDD 阶段 | 技能测试 | 成功标准 |
|-----------|---------------|------------------|
| **红色阶段** | 在没有技能的情况下运行场景 | 代理失败，记录合理化解释 |
| **验证红色阶段** | 捕获确切措辞 | 逐字记录失败 |
| **绿色阶段** | 编写技能解决失败 | 代理现在遵从技能 |
| **验证绿色阶段** | 重新测试场景 | 代理在压力下遵循规则 |
| **重构阶段** | 堵塞漏洞 | 为新的合理化解释添加对策 |
| **保持绿色阶段** | 重新验证 | 重构后代理仍然遵从 |

## 底线

**技能创建就是 TDD。同样的原则，同样的循环，同样的好处。**

如果你不会在没有测试的情况下编写代码，那就不要在没有对代理进行测试的情况下编写技能。

文档的红色-绿色-重构与代码的红色-绿色-重构完全一样。

## 实际影响

将 TDD 应用于 TDD 技能本身（2025-10-03）：
- 6 次红色-绿色-重构迭代达到牢不可破
- 基线测试揭示了 10 多种独特的合理化解释
- 每次重构关闭了特定的漏洞
- 最终验证绿色阶段：在最大压力下 100% 遵从
- 同样的流程适用于任何强制执行纪律的技能
